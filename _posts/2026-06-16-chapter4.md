## 4장: 핵심 구현 결정

### 4.1 프로그래밍 언어 선택

- 구현에 들어가면 프로그래밍 언어 안에서 대부분의 시간을 보내게 됨
- 언어 선택은 생산성과 품질에 영향을 줌
  - 익숙한 언어를 쓰는 프로그래머가 낯선 언어를 쓰는 프로그래머보다 더 생산적임
  - 고수준 언어는 저수준 언어보다 더 많은 개념을 적은 코드로 표현하게 해 줌
  - 언어가 제공하는 표현 수단은 프로그래머가 문제를 생각하고 표현하는 방식에도 영향을 줌
- C++로 Fortran식 코드를 쓰는 것처럼, 새 언어를 쓰더라도 오래된 사고방식에 갇힐 수 있음
- 언어마다 장점과 약점이 있음. 사용하는 언어의 특성을 알고 그에 맞게 보완해야 함

#### 언어별 특징

- **Ada**
  - Pascal 기반의 범용 고수준 언어
  - 실시간, 임베디드, 군사, 우주, 항공 전자 시스템에 적합
  - 데이터 추상화와 정보 은닉 강조
- **Assembly**
  - 각 문장이 기계 명령 하나에 대응되는 저수준 언어
  - 실행 속도나 코드 크기 한계를 밀어붙여야 할 때 사용됨
- **C**
  - UNIX와 함께 성장한 범용 중간 수준 언어
  - 구조화 데이터와 제어 흐름을 제공하지만 포인터, 주소, 비트 조작 등 저수준 성격도 강함
- **C++**
  - C 기반 객체지향 언어
  - 클래스, 다형성, 예외 처리, 템플릿, 표준 라이브러리 제공
- **C#**
  - Microsoft가 만든 범용 객체지향 언어와 개발 환경
  - C, C++, Java와 비슷한 문법을 가지며 Microsoft 플랫폼 개발 도구 지원이 강함
- **Cobol**
  - 비즈니스 애플리케이션 중심의 영어식 언어
  - 오래되었지만 여전히 널리 사용됨
- **Fortran**
  - 최초의 고수준 언어
  - 과학, 공학 계산 분야에서 주로 사용됨
- **Java**
  - C, C++와 비슷한 문법의 객체지향 언어
  - 바이트코드와 가상 머신을 통해 여러 플랫폼에서 실행되도록 설계됨
- **JavaScript**
  - 주로 웹 페이지의 클라이언트 측 동작을 위해 쓰이는 인터프리터 스크립트 언어
- **Perl**
  - 문자열 처리와 시스템 관리 작업에 강한 스크립트 언어
- **PHP**
  - 서버 측 웹 페이지와 데이터베이스 연동에 많이 쓰이는 오픈소스 스크립트 언어
- **Python**
  - 여러 환경에서 동작하는 인터프리터, 대화형, 객체지향 언어
  - 스크립트와 작은 웹 애플리케이션에 자주 쓰임
- **SQL**
  - 관계형 데이터베이스 질의, 갱신, 관리에 쓰이는 선언형 언어
- **Visual Basic**
  - Basic에서 발전한 Microsoft의 고수준 객체지향 시각적 프로그래밍 언어

### 4.2 프로그래밍 규약

- 고품질 소프트웨어에는 아키텍처의 개념적 일관성과 저수준 구현의 일관성이 함께 필요함
- 변수명, 클래스명, 루틴명, 포맷팅, 주석 규약은 구현의 저수준 조화를 만드는 장치임
- 큰 프로그램은 언어 수준의 세부사항을 통제하는 구조가 필요함
- 임의적인 스타일 차이는 이해 비용만 늘림. 뇌는 실제 문제의 차이에 집중해야 함
- 좋은 전체 설계가 있어도 구현 일부가 서로 다른 양식으로 뒤섞이면 개념적 무결성이 깨짐
- 구현이 시작되기 전에 프로그래밍 규약을 정해야 함. 작성된 뒤에 맞추기는 거의 불가능함

### 4.3 기술 파도 위의 위치

- 기술은 파도처럼 초기, 성장, 성숙 단계를 거침
- 성숙한 기술 환경에서는 언어 선택지, 디버거, 최적화 도구, 문서, 사례, 교육 자료가 풍부함
- 초기 기술 환경에서는 언어와 도구가 불안정하고 문서가 부족함
  - 언어 동작을 알아내는 데 시간이 듦
  - 라이브러리나 도구 결함을 우회해야 함
  - 버전업 때 기존 코드가 깨지기 쉬움
  - 서로 다른 도구의 통합이 약함
- 초기 기술을 피하라는 뜻은 아님. 혁신적인 애플리케이션은 초기 기술에서 나오기도 함
- 핵심은 현재 기술 파도에서 어디에 있는지 알고 기대와 계획을 조정하는 것임

#### 언어 안에서 프로그래밍하기와 언어로 표현하기

- 언어 안에서 프로그래밍하면, 언어가 직접 제공하는 구성 요소에 사고가 제한됨
- 언어로 표현하기는 먼저 표현하려는 설계 생각을 정하고, 그 생각을 현재 언어의 도구로 구현하는 방식임
- 언어가 원하는 구조를 직접 지원하지 않아도 규약, 표준, 클래스 라이브러리, 보조 도구로 보완할 수 있음
- 중요한 프로그래밍 원칙 대부분은 특정 언어보다 그 언어를 사용하는 방식에 달려 있음

#### Visual Basic 사례

- 초기 Visual Basic은 비즈니스 로직, UI, 데이터베이스를 분리하도록 강하게 지원하지 않았음
- form 파일은 데이터 저장과 읽기만 담당하고, 비즈니스 로직은 별도 파일에 두는 규약을 둠
- 모든 form이 같은 방식으로 완료 여부를 알려 주게 해서 의미를 단순하게 유지함
- 언어가 구조를 직접 지원하지 않았지만, 간단한 규약으로 프로젝트를 지적으로 관리 가능하게 만듦

### 4.4 주요 구현 실천법 선택

- 구현 준비에는 어떤 실천법을 강조할지 의식적으로 정하는 일이 포함됨
- 페어 프로그래밍과 테스트 우선 개발을 쓸 수도 있고, 개인 개발과 공식 검사를 쓸 수도 있음
- 어느 조합이든 프로젝트 상황에 맞으면 효과적일 수 있음

### 요점 정리

- 모든 프로그래밍 언어에는 강점과 약점이 있다. 사용하는 언어의 특성을 의식해야 한다.
- 프로그래밍을 시작하기 전에 프로그래밍 규약을 정해야 한다. 나중에 기존 코드를 규약에 맞추기는 거의 불가능하다.
- 한 프로젝트에서 사용할 수 있는 구현 실천법은 제한되어 있다. 프로젝트에 가장 잘 맞는 실천법을 의식적으로 선택해야 한다.
- 사용하는 프로그래밍 실천법이 언어에 대한 대응인지, 언어에 의해 통제되는 것인지 점검해야 한다. 언어 안에서만 프로그래밍하지 말고 언어로 표현해야 한다.
- 기술 파도 위의 위치는 효과적이거나 가능한 접근 방식을 결정한다. 현재 위치를 파악하고 계획과 기대를 조정해야 한다.
