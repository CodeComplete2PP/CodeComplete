## 7장: 고급 루틴

### 7.1 루틴을 작성하는 이유

- 루틴은 하나의 목적을 위해 호출할 수 있는 개별 메서드나 절차임
- 좋은 루틴은 프로그램을 읽고 이해하기 쉽게 만드는 핵심 도구임
- 루틴을 만드는 이유는 중복 제거만이 아님

#### 유효한 이유

- **복잡성 감소**
  - 루틴을 만들어 세부사항을 숨기고, 호출하는 쪽에서는 그 내부를 잊을 수 있게 함
  - 깊게 중첩된 루프나 조건문은 별도 루틴으로 빼면 복잡성이 줄어듦
- **이해 가능한 중간 추상화 도입**
  - 여러 줄의 세부 코드를 잘 이름 붙인 루틴 호출 하나로 바꾸면 의도가 분명해짐
- **중복 코드 제거**
  - 같은 코드가 여러 곳에 있으면 수정과 검토가 어려워지고, 서로 다르게 고칠 위험이 커짐
- **하위 클래스 작성 지원**
  - 짧고 잘 나뉜 override 가능 루틴은 하위 클래스에서 안전하게 재정의하기 쉬움
- **순서 숨기기**
  - 여러 작업이 특정 순서로 수행되어야 한다는 가정을 한 루틴 안에 가둠
- **포인터 연산 숨기기**
  - 읽기 어렵고 오류가 많은 포인터 조작은 별도 루틴으로 격리함
- **이식성 향상**
  - 비표준 언어 기능, 하드웨어, 운영체제 의존성을 루틴 안에 모음
- **복잡한 boolean 테스트 단순화**
  - 조건의 세부사항보다 조건의 의미를 함수 이름으로 드러냄
- **성능 향상**
  - 최적화할 코드를 한 곳에 모으면 프로파일링과 개선이 쉬워짐

#### 너무 단순해 보이는 작업도 루틴이 될 수 있음

- 두세 줄짜리 코드라도 이름이 붙으면 읽기 쉬워질 수 있음
- 작은 계산도 나중에 예외 처리나 추가 조건이 생기면 커질 수 있음
- 작은 루틴은 중복된 수정 비용을 줄이는 보험 역할을 함

#### 루틴 생성 이유 요약

- 복잡성 감소
- 이해 가능한 중간 추상화 도입
- 중복 코드 제거
- 하위 클래스 작성 지원
- 순서 숨기기
- 포인터 연산 숨기기
- 이식성 향상
- 복잡한 boolean 테스트 단순화
- 성능 향상
- 복잡성 격리
- 구현 세부사항 숨기기
- 변경 영향 제한
- 전역 데이터 숨기기
- 제어 지점 중앙화
- 재사용 코드 촉진
- 특정 리팩터링 수행

### 7.2 루틴 수준의 설계

- 루틴 수준에서는 응집도가 핵심 설계 휴리스틱임
- 응집도는 루틴 안의 작업들이 서로 얼마나 밀접하게 관련되는지를 뜻함
- 목표는 루틴이 한 가지 일을 잘하고 다른 일을 하지 않게 만드는 것임

#### 기능적 응집도

- 가장 강하고 바람직한 응집도임
- 루틴이 하나의 작업만 수행할 때 생김
- `GetCustomerName()`, `EraseFile()`, `CalculateLoanPayment()` 같은 이름은 실제 구현도 이름과 일치할 때 기능적 응집도를 가짐

#### 덜 이상적인 응집도

- **순차적 응집도**
  - 작업들이 특정 순서로 수행되고, 앞 단계 결과가 다음 단계에 사용됨
  - 각 작업을 별도 루틴으로 나누고, 필요하면 한 루틴이 다른 루틴을 호출하게 함
- **통신적 응집도**
  - 작업들이 같은 데이터를 사용한다는 이유로만 묶임
  - 보고서 출력과 데이터 초기화처럼 목적이 다르면 분리하는 편이 좋음
- **시간적 응집도**
  - 시작, 종료처럼 같은 시점에 수행된다는 이유로 묶임
  - 직접 모든 작업을 수행하기보다 구체 작업 루틴들을 조율하는 역할로 두면 더 낫음

#### 피해야 할 응집도

- **절차적 응집도**
  - 작업들이 특정 순서로 나열될 뿐 완성된 한 기능을 이루지 못함
  - 각 작업을 개별 루틴으로 나누고 호출 루틴도 하나의 완성된 목적을 갖게 정리함
- **논리적 응집도**
  - 여러 작업을 한 루틴에 넣고 플래그로 수행할 작업을 고름
  - 플래그로 다른 루틴의 처리를 제어하기보다 작업별 루틴을 따로 둠
  - 단순 이벤트 핸들러처럼 분기와 위임만 수행하는 경우는 예외적으로 받아들일 수 있음
- **우연적 응집도**
  - 작업들 사이에 식별 가능한 관계가 없음
  - 보통 부분 수정이 아니라 더 깊은 재설계가 필요함

### 7.3 좋은 루틴 이름

- 좋은 루틴 이름은 루틴이 하는 모든 일을 명확하게 설명함
- 이름에 모든 출력과 부작용이 드러나야 함
- 이름이 너무 길고 우스워진다면 이름을 줄일 문제가 아니라 루틴의 부작용과 책임을 줄일 문제일 수 있음
- 의미 없는 동사를 피함
  - `Handle`, `Perform`, `Process`, `DealWith` 같은 이름은 구체적 의미가 약함
  - 단, 이벤트 처리의 기술적 의미로 `Handle`을 쓰는 경우는 예외가 될 수 있음
- 번호만으로 루틴 이름을 구분하지 않음
  - `OutputUser1`, `OutputUser2`는 서로 다른 추상화를 설명하지 못함
- 이름은 이해에 필요한 만큼 길게 지음
- 함수 이름은 반환값을 설명함
  - `IsReady()`, `CurrentColor()`처럼 무엇을 돌려주는지 드러냄
- 프로시저 이름은 강한 동사와 객체를 사용함
  - 객체지향 언어에서는 객체 이름이 호출식에 포함되므로 루틴 이름에서 중복하지 않음
- 반대어를 정확하게 맞춤
  - add/remove, open/close, create/destroy, start/stop, get/set처럼 일관된 쌍을 사용함
- 공통 작업에는 명명 규약을 둠
  - 같은 의미의 ID 조회가 클래스마다 `GetId()`, `id()`, 기본 반환값처럼 흩어지면 기억 비용이 커짐

### 7.4 루틴의 길이에 대한 문제

- 루틴 길이에 대한 절대 기준은 적합하지 않음
- 한 화면 또는 50~150줄 같은 기준이 자주 언급되지만, 연구 결과는 단순히 짧을수록 좋다고 말하지 않음
- 객체지향 프로그램에는 아주 짧은 접근자와 드물게 긴 알고리즘 루틴이 함께 존재함
- 복잡한 알고리즘은 자연스럽게 100~200줄까지 커질 수 있음
- 길이 자체보다 다음 요소가 더 중요함
  - 응집도
  - 중첩 깊이
  - 변수 수
  - 분기 수
  - 필요한 주석 양
  - 기타 복잡성 관련 요소
- 200줄을 넘기려면 신중해야 함. 이해 가능성의 한계에 부딪힐 가능성이 커짐

### 7.5 루틴 매개변수 처리

- 루틴 간 인터페이스는 오류가 많이 생기는 영역임
- 매개변수 설계는 루틴 품질과 직접 연결됨

#### 매개변수 순서와 표시

- 매개변수는 입력, 수정, 출력 순서로 배치함
- 프로젝트에 다른 일관된 규약이 있다면 그 규약을 따르는 것도 도움이 됨
- 언어가 `in`, `out`을 지원하지 않으면 규약이나 타입 시스템으로 역할을 드러낼 수 있음
- 비슷한 루틴들이 비슷한 매개변수를 쓴다면 순서를 일관되게 유지함
- 상태나 오류 변수는 보통 마지막에 둠

#### 매개변수 사용 원칙

- 전달한 매개변수는 모두 사용함
- 입력 매개변수를 작업 변수로 사용하지 않음
  - 원래 입력값이 필요할 때 혼동이 생기므로 별도 지역 변수를 둠
- 매개변수에 대한 인터페이스 가정을 문서화하거나 assertion으로 표현함
  - 입력/수정/출력 여부
  - 수치 단위
  - 상태 코드와 오류값의 의미
  - 예상 값 범위
  - 절대 나오면 안 되는 값
- 매개변수 수는 대략 7개 이하로 제한함
- 같은 데이터가 많은 루틴에 반복 전달되면 루틴을 클래스로 묶고 그 데이터를 객체 데이터로 둘 수 있는지 검토함

#### 객체 전체와 개별 값 사이의 선택

- 루틴 인터페이스가 어떤 추상화를 제공하는지가 기준임
- 루틴이 특정 세 값만 필요로 한다는 추상화라면 그 값들을 개별로 넘김
- 루틴이 해당 객체를 대상으로 작업한다는 추상화라면 객체 전체를 넘김
- 호출 전후에 객체를 억지로 채우거나 다시 해체하는 코드가 필요하면 인터페이스 설계가 잘못됐을 가능성이 있음
- 매번 같은 객체에서 온 매개변수 목록이 바뀐다면 객체 전체를 넘기는 편이 더 안정적일 수 있음

#### 기타 지침

- 언어가 지원하면 named parameter를 사용해 실매개변수와 형식매개변수 대응을 분명히 함
- 형식매개변수와 실매개변수의 타입이 맞는지 확인함
- 컴파일러 경고를 무시하지 않음

### 7.6 함수를 사용할 때 특별히 고려해야 할 사항

#### 함수와 프로시저 선택

- 함수는 값을 반환하는 루틴이고, 프로시저는 값을 반환하지 않는 루틴임
- 의미가 문법보다 중요함
- 함수는 이름이 설명하는 값을 반환하는 것이 주목적일 때 사용함
- 루틴의 주목적이 동작 수행이라면 프로시저가 더 적합함
- 성공/실패 상태를 함수 반환값으로 돌려주는 방식은 일관되게 쓰면 가능하지만, 호출과 상태 검사를 분리하는 편이 더 읽기 쉬울 수 있음

#### 함수 반환값 설정

- 모든 실행 경로에서 유효한 값을 반환하는지 확인함
- 함수 시작 시 기본 반환값을 초기화하면 일부 경로에서 반환값 설정을 빠뜨리는 위험을 줄일 수 있음
- 지역 데이터에 대한 포인터나 참조를 반환하지 않음
- 객체 내부 데이터가 필요하면 멤버 데이터로 저장하고 접근 루틴을 제공함

### 7.7 매크로 루틴과 인라인 루틴

#### 매크로 루틴

- 매크로는 전처리 단계에서 코드로 확장되므로 일반 루틴보다 위험함
- 매크로 표현식은 전체와 각 인수를 괄호로 감싸야 함
- 여러 문장으로 확장되는 매크로는 하나의 블록처럼 동작하게 감싸야 함
- 루틴처럼 대체될 수 있는 매크로는 나중에 루틴으로 바꿔도 호출부가 흔들리지 않게 이름을 정함
- 현대 C++에서는 많은 매크로 용도를 `const`, `inline`, `template`, `enum`, `typedef` 같은 대안으로 대체할 수 있음
- 조건부 컴파일 같은 경우를 제외하면 매크로를 루틴 대안으로 쓰는 일은 최후의 수단에 가까움

#### 인라인 루틴

- 인라인 루틴은 작성 시에는 루틴처럼 보이지만 컴파일 시 호출 지점에 코드가 펼쳐질 수 있음
- C++에서는 구현 코드가 헤더에 노출되어 캡슐화를 약화시킬 수 있음
- 루틴 크기에 따라 코드 크기가 증가할 수 있음
- 성능 목적의 인라인은 프로파일링과 측정으로 이득을 확인한 뒤 사용함

### 요점 정리

- 루틴을 만드는 가장 중요한 이유는 프로그램을 지적으로 관리하기 쉽게 만드는 것이다. 공간 절약은 부차적 이유이고, 가독성, 신뢰성, 수정 용이성이 더 좋은 이유이다.
- 때로는 매우 단순한 작업이 별도 루틴으로 분리될 때 가장 큰 이익을 준다.
- 루틴은 여러 응집도 유형으로 분류할 수 있지만, 대부분의 루틴은 가장 좋은 형태인 기능적 응집도를 갖게 만들 수 있다.
- 루틴 이름은 품질의 신호이다. 이름이 나쁘고 정확하다면 루틴 설계가 나쁠 수 있고, 이름이 나쁘고 부정확하다면 프로그램이 하는 일을 말해 주지 못한다.
- 함수는 함수 이름이 설명하는 특정 값을 반환하는 것이 주목적일 때만 사용해야 한다.
- 신중한 프로그래머는 매크로 루틴을 조심스럽게, 최후의 수단으로만 사용한다.
