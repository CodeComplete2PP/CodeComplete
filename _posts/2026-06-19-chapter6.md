## 6장: 클래스 다루기

### 6.1 클래스의 토대: 추상 데이터형(ADT)

- 클래스는 응집력 있고 잘 정의된 책임을 공유하는 데이터와 루틴의 묶음임
- 한 코드 영역을 작업할 때 안전하게 무시할 수 있는 프로그램 영역을 늘리는 것이 핵심임
- 클래스는 그 목적을 달성하기 위한 주요 도구임
- ADT는 데이터와 그 데이터에 작용하는 연산의 묶음임
- ADT를 이해하지 못하면 클래스가 관련 없는 데이터와 루틴을 담는 편의 상자에 머무르기 쉬움
- ADT를 잘 쓰면 구현 세부사항보다 문제 도메인의 개념으로 작업할 수 있음

#### ADT 사용의 이점

- 구현 세부사항을 숨길 수 있음
  - 데이터 표현이 바뀌어도 프로그램 전체를 수정하지 않아도 됨
- 변경 영향이 프로그램 전체로 퍼지지 않음
  - 새 기능이나 표현 변경을 한 영역에 제한할 수 있음
- 인터페이스가 더 명확해짐
  - `size = 16`보다 `SetSizeInPoints()`처럼 의도를 드러낼 수 있음
- 성능 개선이 쉬워짐
  - 잘 정의된 루틴 몇 개만 다시 구현하면 됨
- 프로그램의 정확성을 확인하기 쉬워짐
  - 비트 플래그 조작보다 의미 있는 루틴 호출이 검토하기 쉬움
- 코드가 더 자기 설명적이 됨
- 데이터를 프로그램 전체에 전달하지 않아도 됨
- 저수준 구현 구조보다 실제 세계의 개념으로 작업할 수 있음

#### ADT 사용 지침

- 스택, 리스트, 큐 같은 저수준 데이터 타입도 ADT로 다룸
- 자료구조 이름보다 그 자료구조가 표현하는 실제 개념을 우선함
  - 직원 목록이면 `List`보다 `Employees` 추상화가 더 적합함
- 파일 같은 흔한 객체도 ADT로 생각함
- 단순한 항목도 ADT로 다루면 이후 변경을 흡수하기 쉬움
- ADT는 클래스 설계의 출발점임

### 6.2 좋은 클래스 인터페이스

#### 좋은 추상화

- 클래스 인터페이스는 일관된 추상화 수준을 제공해야 함
- 각 클래스는 하나의 ADT만 구현해야 함
- 컨테이너나 라이브러리 클래스를 내부에서 사용하더라도 그 사실을 인터페이스에 흘리면 안 됨
- 어떤 추상화를 구현하는지 먼저 분명히 이해해야 함
  - 스프레드시트 컨트롤을 내부 구현으로 쓰더라도, 필요한 추상화가 그리드라면 그리드 인터페이스만 노출해야 함
- 작업은 가능한 한 대응되는 반대 작업과 함께 검토함
  - 켜기/끄기, 추가/삭제, 활성화/비활성화처럼 짝이 필요한지 확인함
- 관련 없는 정보는 다른 클래스로 옮김
- 인터페이스는 가능한 한 의미적 계약보다 프로그램적으로 강제되는 계약에 가깝게 만듦
  - 호출 순서나 초기화 조건처럼 컴파일러가 강제하지 못하는 가정은 오용되기 쉬움
- 수정 중 인터페이스 추상화가 침식되지 않게 주의함
- 새 public 멤버를 추가할 때 기존 인터페이스 추상화와 맞는지 확인함
- 추상화와 응집도는 함께 봐야 함

#### 좋은 캡슐화

- 캡슐화는 추상화를 지키는 장치임
- 클래스와 멤버의 접근 가능성을 최소화함
- 공개 인터페이스에 멤버 데이터를 노출하지 않음
- 언어가 허용하는 한 구현 세부사항을 인터페이스에서 숨김
- 클래스 사용자를 지나치게 가정하지 않음
- friend 클래스는 특별히 통제된 경우가 아니면 피함
- 어떤 루틴이 public 루틴만 사용한다고 해서 public 인터페이스에 넣을 이유가 되지는 않음
- 작성 편의보다 읽기 편의를 우선함
- 구현을 봐야만 클래스를 쓸 수 있다면 캡슐화가 깨진 것임
- 클래스 사용법이 인터페이스만으로 이해되지 않으면 구현을 뒤지는 대신 인터페이스 문서를 고쳐야 함
- 결합이 너무 강한지 계속 확인함
  - 접근성 최소화, 데이터 은닉, 의미적 캡슐화 유지, Demeter 법칙 준수가 모두 결합 완화와 연결됨

### 6.3 설계와 구현 문제

#### 포함: has-a 관계

- 포함은 클래스가 다른 데이터나 객체를 멤버로 갖는 방식임
- 객체지향에서 가장 실용적인 기본 기법에 가까움
- `has-a` 관계는 보통 포함으로 구현함
- private 상속으로 포함을 흉내 내는 방식은 최후의 수단임
- 데이터 멤버가 대략 7개를 넘으면 클래스를 더 작은 클래스로 나눌 수 있는지 검토함

#### 상속: is-a 관계

- 상속은 한 클래스가 다른 클래스의 특수한 형태일 때 사용함
- public 상속은 `is-a` 관계를 의미해야 함
- 기반 클래스의 인터페이스 계약을 완전히 지킬 수 없다면 상속은 맞지 않음
- 상속을 허용하려면 상속을 위해 설계하고 문서화해야 함. 그렇지 않으면 금지하는 편이 낫음
- Liskov Substitution Principle을 지켜야 함
  - 하위 클래스는 기반 클래스 인터페이스로 사용해도 의미가 달라지지 않아야 함
- 인터페이스 상속과 구현 상속을 혼동하지 않음
- 구현만 쓰고 싶다면 상속보다 포함을 사용함
- override할 수 없는 멤버와 같은 이름을 하위 클래스에서 다시 쓰지 않음
- 공통 인터페이스, 데이터, 동작은 가능한 한 상속 트리 위쪽으로 올리되 추상화를 깨면 안 됨
- 인스턴스가 하나뿐인 클래스와 하위 클래스가 하나뿐인 기반 클래스는 의심함
- 하위 클래스에서 루틴을 override하고 아무 일도 하지 않는 구조는 기반 추상화가 잘못됐을 가능성이 큼
- 깊은 상속 트리는 복잡도와 결함 가능성을 높임
- 반복되는 타입 검사는 다형성으로 바꿀 수 있는지 검토함
- 기반 클래스의 데이터는 protected보다 private으로 두는 편이 좋음
- 다중 상속은 복잡도 비용이 크므로 신중하게 사용함
- 상속은 복잡성을 줄일 때만 가치가 있음. 기본 태도는 상속에 보수적인 쪽이 좋음

#### 멤버 함수와 데이터

- 클래스 안의 루틴 수는 가능한 한 작게 유지함
- 원하지 않는 암시적 생성자, 대입 연산자, 기타 연산은 막음
- 클래스가 호출하는 다른 루틴의 종류와 수를 줄임
- 다른 클래스에 대한 간접 호출을 최소화함
  - `account.ContactPerson().DaytimeContactInfo().PhoneNumber()` 같은 호출은 결합을 키움
- 클래스가 다른 클래스와 협력하는 범위를 꼭 필요한 수준으로 제한함

#### 생성자

- 가능하면 모든 생성자에서 모든 멤버 데이터를 초기화함
- singleton이 필요하면 private 생성자와 단일 접근 루틴으로 강제할 수 있음
- 성능상 측정된 이유가 없다면 얕은 복사보다 깊은 복사를 우선함
- 얕은 복사는 참조 계수, 안전한 복사, 비교, 삭제 등 추가 복잡도를 만든다.

### 6.4 클래스를 작성하는 이유

- 실제 세계 객체를 모델링함
- 추상 객체를 모델링함
- 복잡성을 줄임
- 복잡한 알고리즘, 큰 데이터 집합, 통신 프로토콜 같은 복잡성을 격리함
- 구현 세부사항을 숨김
- 변경 영향을 제한함
- 전역 데이터를 클래스 인터페이스 뒤에 숨김
- 여러 루틴 사이로 반복 전달되는 매개변수를 줄임
- 제어 지점을 중앙화함
- 재사용 가능한 코드를 만들기 쉽게 함
- 프로그램 계열을 고려해 변경될 부분을 분리함
- 관련 연산을 함께 묶음
- 특정 리팩터링을 수행함

#### 피해야 할 클래스

- 모든 것을 알고 모든 것을 제어하는 god class를 피함
- 데이터만 있고 동작이 없는 클래스는 정말 클래스가 맞는지 검토함
- 동작만 있고 데이터가 없는 동사형 클래스는 다른 클래스의 루틴으로 옮길 수 있는지 확인함

### 6.5 프로그래밍 언어와 관련된 이슈

- 언어마다 클래스와 상속을 다루는 방식이 다름
- override 가능 여부, 생성자와 소멸자의 동작, 예외 상황의 생성/소멸 처리, 기본 생성자의 중요도, finalizer 호출 시점, 연산자 override, 메모리 관리 방식은 언어별로 확인해야 함
- 클래스 설계 원칙은 공통적이지만 세부 구현은 사용하는 언어의 규칙을 따라야 함

### 6.6 클래스를 넘어서: 패키지

- 클래스는 모듈성을 얻기 위한 강력한 도구지만 모듈성은 클래스보다 큰 단위도 포함함
- 문장, 루틴, 클래스에 이어 더 큰 집합인 패키지를 다루는 방향으로 발전해 왔음
- 패키지는 여러 객체를 묶어 추상화와 캡슐화를 더 높은 수준에서 지원함
- 언어가 패키지를 직접 지원하지 않으면 규약으로 비슷한 효과를 만들 수 있음
  - public 클래스와 패키지 내부 클래스를 구분하는 이름 규약
  - 패키지 소속을 드러내는 이름 또는 프로젝트 구조 규약
  - 어떤 패키지가 어떤 패키지를 어떤 방식으로 사용할 수 있는지에 대한 규칙

### 참고 자료

#### 클래스 일반

- **Bertrand Meyer, _Object-Oriented Software Construction, 2nd ed._**

  ADT가 클래스의 토대가 되는 방식을 깊게 설명하고, 상속도 자세히 다루는 자료.

- **Arthur J. Riel, _Object-Oriented Design Heuristics_**

  클래스 수준 설계 개선을 위한 실용적인 휴리스틱을 많이 제공하는 자료.

#### C++

- **Scott Meyers, _Effective C++: 50 Specific Ways to Improve Your Programs and Designs, 2nd ed._**

  C++ 클래스와 객체 설계에서 널리 참고되는 실전 지침.

- **Scott Meyers, _More Effective C++: 35 New Ways to Improve Your Programs and Designs_**

  C++의 세부 규칙과 설계 함정을 더 깊게 다루는 자료.

#### Java

- **Joshua Bloch, _Effective Java Programming Language Guide_**

  Java에 특화된 조언과 일반적인 객체지향 실천법을 함께 제공하는 자료.

#### Visual Basic

- **James Foxall, _Practical Standards for Microsoft Visual Basic .NET_**

  Visual Basic .NET 실무 표준을 다루는 자료.

- **Gary Cornell, Jonathan Morrison, _Programming VB .NET_**

  경험 있는 프로그래머를 위한 VB .NET 안내서.

- **Fred Barwell et al., _Professional VB.NET, 2nd ed._**

  VB.NET 개발을 폭넓게 다루는 참고 자료.

### 요점 정리

- 클래스 인터페이스는 일관된 추상화를 제공해야 한다. 많은 문제는 이 원칙을 어길 때 생긴다.
- 클래스 인터페이스는 무언가를 숨겨야 한다. 시스템 인터페이스, 설계 결정, 구현 세부사항이 그 대상이 될 수 있다.
- `is-a` 관계를 모델링하는 경우가 아니라면 상속보다 포함이 대체로 낫다.
- 상속은 유용하지만 복잡성을 추가한다. 이는 복잡성 관리라는 소프트웨어의 핵심 기술 명령과 충돌할 수 있다.
- 클래스는 복잡성을 관리하는 주요 도구이다. 그 목적을 달성할 만큼 충분히 설계에 주의를 기울여야 한다.
