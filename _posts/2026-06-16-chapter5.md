## 5장: 구현에서의 설계

### 5.1 설계의 어려움

- 설계는 요구사항을 동작하는 소프트웨어로 바꾸기 위한 구상, 발명, 고안 활동임
- 요구사항과 코딩, 디버깅을 연결하는 활동임
- 좋은 상위 설계는 여러 하위 설계를 담을 수 있는 구조를 제공함
- 작은 프로젝트에도 설계는 유용하고, 큰 프로젝트에서는 필수임

#### 설계는 까다로운 문제임

- 까다로운 문제는 문제를 풀어 보거나 일부를 풀어 봐야 제대로 정의되는 문제임
- 소프트웨어 설계는 먼저 한 번 풀어야 문제를 명확히 알 수 있고, 다시 풀어야 제대로 동작하는 해법을 만들 수 있음
- 학교 과제는 대개 이런 성격이 약하지만, 실무 프로그램은 요구와 조건이 계속 드러나고 바뀜

#### 설계 과정은 지저분함

- 완성된 설계는 깔끔해야 하지만, 설계 과정은 깔끔하지 않음
- 설계에서는 많은 잘못된 길과 실수가 생김
- 같은 실수를 코드 작성 뒤에 발견하는 것보다 설계 단계에서 발견하고 고치는 편이 비용이 훨씬 낮음
- 설계가 충분한지 판단하기 어렵기 때문에 반복과 검토가 필요함

#### 설계는 절충과 우선순위의 문제임

- 이상적으로는 빠르고, 작고, 오류 없고, 비용도 낮은 시스템을 원함
- 실제로는 속도, 저장 공간, 네트워크 사용량, 개발 시간, 신뢰성 같은 목표가 충돌함
- 설계자는 경쟁하는 특성 사이에서 프로젝트에 맞는 균형을 잡아야 함

#### 설계는 제한을 만듦

- 설계는 가능성을 열기도 하지만, 일부 가능성을 의도적으로 제한하기도 함
- 제한이 없으면 소프트웨어는 필요 이상으로 비대하고 흐트러진 구조가 되기 쉬움
- 좋은 제한은 해법을 단순하게 만들고 결과를 개선함

#### 설계는 비결정적이고 휴리스틱함

- 같은 프로그램을 세 사람이 설계하면 서로 다른 설계가 나올 수 있고, 모두 받아들일 만할 수 있음
- 그래서 설계 기법은 항상 같은 결과를 보장하는 절차보다 경험적 규칙에 가까움
- 한 프로젝트에서 잘 통한 도구나 기법이 다음 프로젝트에서도 반드시 통하지는 않음

#### 설계는 발현적으로 만들어짐

- 설계는 머릿속에서 완성된 형태로 한 번에 나오지 않음
- 설계 리뷰, 비공식 토론, 코드 작성 경험, 수정 경험을 거치며 진화함
- 초기 개발 중에도 바뀌고, 이후 버전으로 확장될 때 더 크게 바뀜

### 5.2 핵심 설계 개념

#### 소프트웨어의 핵심 기술 명령: 복잡성 관리

- 소프트웨어 개발의 어려움은 본질적 어려움과 우발적 어려움으로 나눌 수 있음
- 우발적 어려움은 언어 문법, 비대화식 환경, 도구 통합 부족처럼 본질이 아닌 외부 조건에서 생김
- 본질적 어려움은 현실 세계의 복잡한 개념, 예외, 의존성, 정확성 요구에서 생김
- 남아 있는 가장 큰 기술적 문제의 뿌리는 복잡성임
- 소프트웨어의 핵심 기술 명령은 복잡성 관리임
- 목표는 두 가지임
  - 한 번에 다뤄야 하는 본질적 복잡성의 양을 줄임
  - 우발적 복잡성이 불필요하게 늘어나지 않게 함

#### 복잡성을 줄이는 방향

- 사람은 전체 프로그램을 한 번에 머리에 담을 수 없음
- 시스템을 서브시스템으로 나누고, 잘 정의된 객체와 패키지로 관심사를 분리해야 함
- 짧은 루틴, 문제 도메인 언어, 높은 추상화 수준은 정신적 부담을 줄임
- 단순한 문제에 복잡한 해법을 쓰거나, 복잡한 문제에 단순하지만 틀린 해법을 쓰는 일을 피해야 함

#### 바람직한 설계 특성

- **최소 복잡성**
  - 영리한 설계보다 이해하기 쉬운 설계가 우선임
  - 한 부분에 집중할 때 다른 대부분을 안전하게 무시할 수 있어야 함
- **유지보수 용이성**
  - 유지보수 프로그래머가 무엇을 궁금해할지 생각하며 설계해야 함
- **느슨한 결합**
  - 클래스와 클래스, 루틴과 루틴 사이의 연결을 최소화함
- **확장성**
  - 가능성 높은 변경이 구조를 크게 흔들지 않아야 함
- **재사용성**
  - 시스템 일부를 다른 시스템에서도 사용할 수 있게 설계함
- **높은 fan-in**
  - 여러 클래스가 공통 유틸리티 클래스를 잘 활용하는 구조를 의미함
- **낮거나 중간 수준의 fan-out**
  - 한 클래스가 너무 많은 클래스를 사용하면 복잡도가 높아짐
- **이식성**
  - 다른 환경으로 옮기기 쉽게 설계함
- **간결성**
  - 필요 없는 부분은 제거함. 추가 코드는 개발, 리뷰, 테스트, 유지보수 비용을 계속 만든다.
- **계층화**
  - 한 수준에서 시스템을 볼 때 일관된 관점을 유지할 수 있어야 함
- **표준 기법 사용**
  - 지나치게 이국적인 설계보다 익숙하고 검증된 접근을 우선함

#### 설계 수준

- **수준 1: 소프트웨어 시스템**
  - 전체 시스템 수준의 설계
  - 곧바로 클래스 설계로 내려가기보다 서브시스템이나 패키지 같은 상위 조합을 먼저 생각하는 편이 좋음
- **수준 2: 서브시스템 또는 패키지 분할**
  - 데이터베이스, 사용자 인터페이스, 비즈니스 규칙, 보고 엔진 같은 주요 서브시스템을 식별함
  - 서브시스템 간 통신 규칙을 제한해야 분할의 이점이 생김
  - 의존 관계는 순환이 없는 구조가 좋음
- **수준 3: 클래스 분할**
  - 각 서브시스템을 구현 가능한 클래스 단위로 나눔
  - 클래스 인터페이스와 외부와의 상호작용을 정함
- **수준 4: 루틴 분할**
  - 각 클래스 안의 공개 루틴과 비공개 루틴을 나눔
  - 루틴을 정의하다 보면 클래스 인터페이스에 대한 이해가 바뀌고, 다시 수준 3으로 돌아갈 수 있음
- **수준 5: 루틴 내부 설계**
  - 의사코드, 알고리즘 선택, 코드 단락 구성 같은 세부 설계
  - 항상 수행되지만 의식적으로 잘 하거나 무의식적으로 나쁘게 할 수 있음

### 5.3 설계 구성 블록: 휴리스틱

- 설계는 비결정적이므로 좋은 휴리스틱을 능숙하게 적용하는 일이 핵심임
- 휴리스틱은 정답을 보장하지 않지만, 좋은 설계 통찰을 얻기 위한 시도 방향을 제공함

#### 실제 세계 객체 찾기

- 객체지향 설계의 출발점은 실제 세계와 합성 객체를 식별하는 것임
- 기본 절차
  - 객체와 속성을 식별함
  - 각 객체에 수행할 수 있는 작업을 정함
  - 각 객체가 다른 객체에 무엇을 할 수 있는지 정함
  - 객체의 어떤 부분을 공개하고 어떤 부분을 숨길지 정함
  - 각 객체의 공개 인터페이스를 정의함
- 이 절차는 순서대로 한 번만 수행되지 않음. 반복하면서 조정됨

#### 일관된 추상화 만들기

- 추상화는 세부사항 일부를 안전하게 무시하고 개념을 다루는 능력임
- 좋은 루틴, 클래스, 패키지 인터페이스는 각각의 수준에서 복잡한 내부를 보지 않아도 되게 함
- 추상화의 주요 이점은 관련 없는 세부사항을 무시하게 해 복잡성을 줄이는 데 있음

#### 구현 세부사항 캡슐화

- 추상화가 "높은 수준에서 보아도 된다"라면, 캡슐화는 "다른 수준으로 들여다보면 안 된다"에 가까움
- 캡슐화는 내부 복잡성을 보지 못하게 해서 복잡성을 관리함
- 잘 캡슐화된 클래스는 외부에 필요한 것만 드러내고 나머지는 숨김

#### 상속은 설계를 단순하게 할 때만 사용

- 상속은 비슷한 객체들 사이의 공통점과 차이점을 표현함
- 일반 타입에 공통 동작을 두고, 세부 타입에 차이를 두면 추상화와 함께 강력하게 동작함
- 잘 쓰면 큰 이점이 있지만, 순진하게 쓰면 큰 손상을 만들 수 있음

#### 비밀 숨기기: 정보 은닉

- 정보 은닉은 구조적 설계와 객체지향 설계의 기반 개념임
- 클래스, 패키지, 루틴은 자신이 숨기는 설계 또는 구현 결정을 기준으로 정의될 수 있음
- 숨길 수 있는 비밀
  - 변경될 가능성이 큰 영역
  - 파일 형식
  - 데이터 타입 구현 방식
  - 오류 영향 범위를 제한해야 하는 영역
- 좋은 인터페이스는 내부 동작을 가능한 한 적게 드러냄
- 정보 은닉은 리터럴 대신 이름 있는 상수를 쓰는 수준부터 데이터 타입, 클래스, 루틴, 서브시스템 설계까지 모두 적용됨

#### 정보 은닉을 막는 장벽

- 정보가 시스템 곳곳에 과도하게 퍼져 있으면 숨기기 어려움
- 순환 의존성은 테스트와 이해를 어렵게 함
- 클래스 데이터를 전역 데이터처럼 오해하면 필요한 캡슐화까지 피하게 됨
- 성능 우려 때문에 은닉을 피하는 것은 대개 성급함. 병목을 측정하기 전에는 모듈식 설계가 이후 최적화에도 유리함

#### 변경 가능성이 큰 영역 식별

- 좋은 설계자는 변경을 예상함
- 절차
  - 변경될 가능성이 큰 항목을 식별함
  - 변경될 항목을 분리함
  - 인터페이스를 변경에 둔감하게 만들어 영향 범위를 내부로 제한함
- 자주 변하는 영역
  - 비즈니스 규칙
  - 하드웨어 의존성
  - 입력과 출력
  - 비표준 언어 기능
  - 어려운 설계와 구현 영역
  - 상태 변수
  - 데이터 크기 제약
- 변경 가능성과 대비 비용을 함께 봐야 함
- 가능성이 높은 변경은 쉽게 수용하게 만들고, 극히 낮은 변경만 넓은 영향을 허용함

#### 결합을 느슨하게 유지

- 결합은 한 클래스나 루틴이 다른 클래스나 루틴과 얼마나 강하게 관련되어 있는지를 뜻함
- 좋은 결합은 작고, 직접적이고, 눈에 보이고, 유연함
- 결합 평가 기준
  - **크기**: 연결 수가 적을수록 좋음
  - **가시성**: 연결이 명확할수록 좋음
  - **유연성**: 호출하거나 교체하기 쉬울수록 좋음
- 결합 종류
  - 단순 데이터 매개변수 결합: 원시 데이터가 매개변수로 전달되는 일반적이고 수용 가능한 결합
  - 단순 객체 결합: 객체를 생성하거나 사용하는 결합
  - 객체 매개변수 결합: 다른 객체를 넘기도록 요구해 더 많은 지식을 강제하는 결합
  - 의미적 결합: 상대 모듈의 내부 동작에 대한 암묵적 지식에 의존하는 위험한 결합
- 클래스와 루틴은 복잡성을 줄이는 지적 도구임. 사용하려면 내부까지 알아야 한다면 역할을 못 하는 것임

#### 공통 설계 패턴 찾기

- 설계 패턴은 흔한 소프트웨어 문제에 대한 준비된 해법의 핵심임
- 패턴은 준비된 추상화를 제공해 복잡성을 줄임
- 패턴은 오래 축적된 해법을 제도화해서 오류를 줄임
- 패턴 이름은 설계 대화를 더 높은 수준으로 올려 의사소통을 빠르게 함
- 대표 패턴
  - Abstract Factory
  - Adapter
  - Bridge
  - Composite
  - Decorator
  - Facade
  - Factory Method
  - Iterator
  - Observer
  - Singleton
  - Strategy
  - Template Method
- 주의할 점은 패턴 끼워 맞추기와 패턴을 써 보고 싶어서 쓰는 기능 과잉임

#### 기타 휴리스틱

- **강한 응집도 추구**
  - 클래스의 루틴이나 루틴 내부 코드가 중심 목적을 얼마나 잘 지원하는지 봄
- **계층 만들기**
  - 한 번에 필요한 수준의 세부사항만 보게 함
- **클래스 계약 공식화**
  - 선조건과 후조건으로 인터페이스 책임을 생각함
- **책임 할당**
  - 각 객체가 무엇을 책임져야 하는지 묻는 방식
- **테스트를 위한 설계**
  - 독립적으로 테스트할 수 있는 구조를 생각하면 인터페이스가 더 명확해짐
- **실패 회피**
  - 성공 사례만 따라 하지 말고 실패 모드를 적극적으로 고려함
- **바인딩 시점 의식**
  - 값을 일찍 묶으면 단순하고, 늦게 묶으면 유연함
- **제어 지점 중앙화**
  - 변경할 만한 비자명한 코드는 한 곳에서 찾고 한 곳에서 바꾸게 함
- **무식한 방법도 고려**
  - 동작하는 단순 해법이 동작하지 않는 우아한 해법보다 나음
- **그림 그리기**
  - 문제를 더 높은 추상화 수준에서 보게 함
- **모듈성 유지**
  - 루틴과 클래스를 입력과 출력이 분명한 블랙박스처럼 만들려는 목표

#### 휴리스틱 사용 지침

- 한 접근 방식에 갇히지 않아야 함
- UML이 풀리지 않으면 글로 쓰고, 작은 테스트 프로그램을 만들고, 완전히 다른 접근을 시도함
- 설계 문제 전체를 한 번에 해결할 필요는 없음
- 아직 정보가 부족한 결정을 억지로 닫지 말고, 더 많은 정보를 얻은 뒤 결정할 수 있음

### 5.4 설계 실천법

#### 반복하기

- 설계는 반복 과정임
- 첫 설계가 괜찮아 보여도 멈추지 않는 편이 좋음
- 두 번째 시도가 첫 번째보다 나은 경우가 많고, 각 시도에서 배운 것이 전체 설계를 개선함
- 상위 수준과 하위 수준을 오가며 큰 그림과 세부 현실을 서로 검증해야 함

#### 분할 정복

- 복잡한 프로그램 전체를 한 번에 머리에 담을 수 없음
- 관심사를 나누고 각 영역을 개별적으로 다룸
- 막히는 영역이 있으면 다른 방식으로 반복함

#### top-down과 bottom-up 설계

- top-down은 높은 추상화 수준에서 시작해 점점 세부사항으로 내려감
- bottom-up은 구체적인 객체와 책임에서 시작해 더 일반적인 구조로 올라감
- top-down은 한 번에 다룰 세부사항을 줄여 줌
- bottom-up은 실제로 필요한 유틸리티와 구체 책임을 일찍 드러냄
- 둘은 경쟁 전략이 아니라 상호 보완 전략임
- 설계는 시행착오를 포함하므로 여러 접근을 시도해야 함

#### 실험적 프로토타이핑

- 특정 설계가 동작할지 구현 세부사항을 알아야 판단할 때가 있음
- 실험적 프로토타이핑은 특정 설계 질문에 답하기 위한 최소한의 폐기용 코드를 작성하는 것임
- 위험하거나 낯선 부분을 낮은 비용으로 확인하는 데 좋음
- 프로토타입 코드는 질문에 답하는 최소 범위에 머물러야 함
- 질문에 답한 뒤에는 폐기해야 함. 프로토타입을 제품 코드로 키우면 위험해짐

#### 협력 설계

- 설계는 혼자만의 활동일 필요가 없음
- 비공식 토론, 리뷰, 워크스루, 페어 설계 같은 방식으로 다른 관점을 얻을 수 있음
- 여러 사람이 설계를 보면 숨은 가정과 빠진 대안을 더 빨리 발견할 수 있음

#### 어느 정도 설계해야 하는가

- 구현이 뻔해 보일 정도까지 설계를 밀고 가는 것이 한 기준임
- 모든 세부사항을 설계하려는 것도, 아무 설계 없이 코딩하는 것도 둘 다 틀림
- 프로젝트 규모, 위험, 팀 경험, 변경 비용에 따라 적절한 설계량이 달라짐

#### 설계 작업 기록하기

- 설계 문서화는 무겁게만 할 필요가 없음
- Wiki, 이메일, 사진, 플립 차트, CRC 카드, UML, 코드 주석 등 상황에 맞는 방법을 섞어 쓸 수 있음
- 핵심은 나중에 왜 그렇게 설계했는지와 무엇을 결정했는지를 찾을 수 있게 하는 것임

### 5.5 인기 방법론에 대한 코멘트

- 소프트웨어 설계 역사에는 서로 극단적인 방법론 주장들이 반복됨
- 모든 세부사항을 코딩 전에 설계하라는 주장도 맞지 않고, 아무 설계도 하지 말라는 주장도 맞지 않음
- 큰 사전 설계의 대안은 무설계가 아니라 작은 사전 설계 또는 충분한 사전 설계임
- 정확히 얼마가 충분한지는 판단 문제지만, 항상 틀린 양은 두 가지임
  - 모든 세부사항을 설계하는 것
  - 아무것도 설계하지 않는 것
- 설계는 까다롭고, 지저분하고, 휴리스틱한 과정으로 다뤄야 함
- 첫 설계에 만족하지 말고, 협력하고, 단순함을 추구하고, 필요할 때 프로토타입을 만들고, 반복해야 함

### 요점 정리

- 소프트웨어의 핵심 기술 명령은 복잡성 관리이다. 단순성에 초점을 둔 설계가 이를 크게 돕는다.
- 단순성은 한 번에 다뤄야 하는 본질적 복잡성을 줄이고, 우발적 복잡성이 불필요하게 퍼지지 않게 하는 두 가지 방식으로 달성된다.
- 설계는 휴리스틱하다. 단일 방법론을 교조적으로 따르는 태도는 창의성과 프로그램 품질을 해친다.
- 좋은 설계는 반복적이다. 더 많은 설계 가능성을 시도할수록 최종 설계가 좋아진다.
- 정보 은닉은 특히 가치 있는 개념이다. "무엇을 숨겨야 하는가?"라는 질문은 많은 어려운 설계 이슈를 정리해 준다.
- 설계에 관한 유용한 자료는 이 책 밖에도 많다. 이 장의 관점은 그중 일부에 불과하다.
